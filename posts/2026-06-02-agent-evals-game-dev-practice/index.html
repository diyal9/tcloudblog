Agent Evals：AI 智能体落地的"驾照考试" —— 从理论到游戏研发实战

AI-Agent 正在从‘对话者’变成‘行动者’。在游戏研发中，我们如何评估一个 Agent 是否合格？本文结合开源 Agent Evals 框架，探讨如何构建游戏行业的 AI 评估体系。

在 AI 辅助编程和自动化运维日益普及的今天，我们正经历从 RAG（检索增强生成） 向 Agentic Workflow（智能体工作流） 的范式转移。

如果说 RAG 解决的是“能不能答对”的问题，那么 Agent 解决的则是“能不能做对”的问题。对于一个能修改代码、操作数据库、甚至调整游戏配置的 Agent 来说，传统的文本匹配评估已经彻底失效。

今天，我想聊聊 Agent Evals（智能体评估），以及它为什么是 AI 真正落地游戏研发的“驾照考试”。

1. 为什么 RAG 的评估标准不够用了？#

在 RAG 时代，我们通常用 RAGAS 或 DeepEval 来衡量：

  • 检索相关性：找到的文档和答案相关吗？
  • 忠实度：回答是否基于文档，没有幻觉？

但在 Agent 场景 下，AI 开始使用工具（Tools）：调用 API、执行 Shell 命令、修改文件。此时，评估变得极其复杂：

  • 过程黑盒：结果对了，但中间是不是执行了 rm -rf？
  • 决策路径：AI 绕了 20 步才完成一个本该 3 步完成的任务，效率合格吗？
  • 鲁棒性：当工具报错时，AI 是崩溃乱改，还是能自我修正？

这就是 Agent Evals 要解决的核心问题：验证 Agent 的行为安全性、决策效率和目标达成率。

2. 业界的探索：三大主流开源框架#

在调研中，我发现了几个非常有启发性的开源项目，它们代表了评估的不同方向：

🌪️ Strands Agents Evals：引入“混沌工程”#

亚马逊开源的 Strands 框架做了一个大胆的设计：在评估时故意给 Agent 返回错误的工具调用结果。

  • 目的：测试 Agent 在工具报错时的“容错率”。
  • 启示：好的 Agent 不仅要能修好代码，还要能在环境不稳定时不崩溃、不乱改。

📊 RagaAI Catalyst：全链路可观测性#

不仅仅是评分，更是追踪（Tracing）。它记录了 Agent 的每一次思考（Thought）和工具调用。

  • 价值：当 Agent 表现不佳时，我们能像看“行车记录仪”一样回溯它的决策过程。

🔌 OpenLIT：标准化的监控协议#

基于 OpenTelemetry 构建，让 Agent 的评估数据可以直接接入 Prometheus/Grafana 等现有监控系统。

3. 结合实战：游戏公司的 AI 落地方案#

回到我们的业务场景。在游戏研发中，引入 AI-Agent 不仅仅是让程序员写代码快一点，更是要构建一套 自动化、高质量的生产管线。

结合上述理论，我设想了一套针对游戏行业的 Agent Evals 落地Architecture：

场景 A：QA Agent 的“行为验证”#

传统自动化测试往往是脚本式的。而 Agent 需要理解自然语言的需求（如“测试签到功能的内存泄漏”）。

  • 评估点：
    • 操作序列：Agent 是否正确地点击了 UI？是否读取了 Memory Profiler？
    • 结果判定：它生成的测试报告是否准确指出了泄漏点，而不是胡编乱造？

场景 B：代码 Agent 的“安全围栏”#

程序员的 AI 助手如果不小心引入了一个 Bug，或者误删了核心配置，代价是巨大的。

  • 评估点：
    • Diff 审计：Agent 提交的代码 Diff 是否符合规范？有没有引入危险的依赖？
    • 影响面分析：修改 PaySDK 是否考虑到了所有调用方？
  • 策略：在 CI/CD 中加入 Agent Evals，如果 Agent 的修改导致核心测试用例失败，或者偏离了“标准操作路径”，直接拒绝合并。

场景 C：资产管线的“一致性检查”#

策划用 AI 生成了复杂的 UI 配置表，或者 AI 自动调整了 Prefab。

  • 评估点：
    • 结构完整性：生成的 JSON 是否符合 Schema？引用的资源 ID 是否存在？
    • 逻辑自洽：配置表中的数值是否在合理范围内？（比如伤害值不能为负）

4. Architecture设计：GOKES (Game Ontology & Knowledge Engineering System)#

为了实现上述评估，我们需要将评估系统集成到更宏大的 知识工程系统 (GOKES) 中。

核心Architecture组件#

  1. Eval Runner (Go)：
    • 作为执行器，并发跑各种测试用例（Golden Paths）。
  2. Trace Collector：
    • 收集 Agent 的全量操作日志，构建“决策树”。
  3. Judge Service：
    • Hard Check：基于 AST 和规则的硬性检查（如：禁止删除 GameManager）。
    • Soft Check：基于 LLM 的软性打分（如：代码风格是否优雅，逻辑是否合理）。
  4. Result Dashboard (Vue Flow)：
    • 可视化展示 Agent 的“实际操作路径”与“标准路径”的偏差，帮助人类快速定位问题。

知识库与Ontology的结合#

这正是我们之前讨论的 Ontology (Ontology) 发挥作用的地方。

  • 知识图谱 提供了代码、需求、测试用例之间的关联。
  • 当 Agent 修改代码时，Evals 系统可以利用图谱自动判断：“这次修改是否覆盖了所有相关的测试用例？”

5. 写在最后#

Agent Evals 不是限制 AI 的枷锁，而是让 AI 跑得更快的安全带。

在游戏研发这样对稳定性要求极高的行业，我们绝不能容忍一个“黑盒”AI 随意修改我们的代码库和资产。只有通过建立完善的评估体系，结合 Harness Engineering 的理念，我们才能真正放心地把任务交给 Agent。

下一步，我将着手开发 CLI 工具的原型，尝试用 Tree-sitter 和 SQLite 搭建这套系统的 MVP。

AI 不会淘汰开发者，但会用 AI 的开发者将淘汰不会用 AI 的开发者。 而 Agent Evals，就是区分这两者的关键分水岭。
