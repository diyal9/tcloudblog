在构建 LLM 和 Agent 系统时，团队常面临一个核心难题：如何判断应用变好了还是变差了？

传统软件测试的是确定性代码路径，输入确定则输出确定。但 AI 输出具有概率性——同一个功能可能因为 prompt 调整、模型版本更新、检索数据变化或工具调用顺序不同而表现迥异。

Evals（评估） 的作用，就是将这些难以捉摸的业务质量、安全边界、任务完成度、格式约束和推理过程，转化为可重复运行的检查机制。

成熟团队不会把 Evals 当成上线前的一次性考试，而是将其嵌入到 AI Engineering Loop（AI 工程循环） 中。对 Agent 尤其如此，因为 Agent 不仅有最终答案，还包含中间轨迹、工具错误、策略偏移和高风险行为。

换句话说，Evals 是 AI 工程里的规格说明、回归测试和质量仪表盘，让团队从“感觉模型变好了”走向“我们知道它在哪些场景变好了，在哪些场景还不可靠”。

1. 核心认知：Evals 是工程机制而非一次性考试#

AI 评估体系的核心价值在于：

  • 把业务质量、安全边界、任务完成度转化为可运行的检查：有些用代码判断，有些由人工标注，有些交给 LLM 裁判。
  • 嵌入完整循环：从生产 Traces 中发现失败模式，沉淀数据集和黄金样例，用 Evals 比较 Prompt 和系统设计改动，再通过 CI 门禁、灰度发布、线上监控和漂移检测持续回流。

2. 三种核心评估方法#

成熟的评估体系通常会组合使用以下三种方法：

人工评估 (Human Evaluation)#

这是所有自动评估的基石。

  • 价值：只有人工阅读输出，才能真正理解应用在具体用例下“什么是好”、“什么是差”。这决定了后续构建哪些自动评估器以及定义什么标准。
  • 产出：产生人工标注，作为验证自动评估器的基准真值 (Ground Truth)。
  • 建议：跳过这一步直接进入自动化的团队，最后往往在衡量无关紧要的东西。人工评估也不是一劳永逸，需持续抽检以保持校准。

基于代码的评估 (Code-based Evaluation)#

使用确定性逻辑验证属性的检查器。

  • 特点：速度快、成本低、结果完全一致。
  • 适用场景：
    • 输出是否为有效 JSON / 符合要求的 Schema。
    • 是否包含（或不包含）特定关键词。
    • 输出长度限制、生成的 SQL 是否可执行等。
  • 局限：无法评估语义。它能检查是否包含 “refund” 这个词，但无法判断是否“正确解释了退款政策”。

LLM-as-a-Judge (模型裁判)#

使用语言模型为输出打分。

  • 适用场景：需要理解语言语义的任务，如相关性、语气匹配、摘要质量、推理过程等。
  • 局限与风险：
    • 缺乏上下文：模型不会自动像人类专家一样打分。
    • 需要校准：必须用人类偏好验证其衡量的内容是否准确。
    • 盲点共享：可能与被评估的应用 LLM 共享盲点（尤其是同一家族模型）。
  • 建议：经过人工标注校准、并由基于代码的检查兜底的 LLM 裁判是可靠的。

3. 评估策略：参考答案 vs 无参考答案#

类型描述优势局限
基于参考答案将输出与预定义的“黄金回答”进行比较标准明确，易于量化需要预先定义参考回答，无法覆盖长尾数据
无参考答案仅评估输出本身（如检查格式、有害内容）可应用到未见过的生产数据上标准定义较难，可能不够精确

4. 实践指南：从哪里开始#

4.1 什么时候设置评估器？#

从人工审阅开始。

  • 如果问题修一次就解决（如简单 Prompt 修改），直接修，不需要评估器。
  • 如果能清楚识别出一个反复出现的失败模式，并且希望在不同输入上测试它，则设置评估器。

4.2 应该评估什么？#

避免使用“有帮助”或“质量”等通用属性，它们产生的信号模糊。

  • 精确定义：明确你的应用里什么算好、什么算差。
  • 使用二元分数 (Pass/Fail)：强制定义可接受和不可接受的边界。分级量表（1-5 分）容易引入歧义，降低一致性。

4.3 组合使用#

  • 代码评估：兜底格式和硬性约束。
  • LLM 评估：检查语义和软性质量。
  • 人工评估：持续抽检，校准上述两者。

5. 闭环流程#

  1. 人工审阅：建立直觉，识别失败模式。
  2. 自动化评估：针对反复出现的模式进行回归测试。
  3. 发布：如果结果通过，发布变更。
  4. 生产监控：上线后，利用无参考答案评估器和用户反馈监控实时流量。
  5. 回流：如果生产中出现新异常，将其捕获为 Traces -> 数据集 -> 下一轮实验。

Evals 的完整闭环是：从感觉到知道，从人工到自动化，从离线到在线。

6. 落地实战：游戏公司的 Evals 建设方案#

了解了 Evals 的三种方法后，结合前文提到的 SDD 上下文工程 和我们的 AI Collab Hub Architecture，我们如何在一个真实的游戏研发项目中落地这套体系？

核心目标是将 Evals 嵌入到现有的 Agent Sandbox 和 Hub 审批流 中，解决"AI 生成代码质量不可控"的痛点。

6.1 评估体系与 SDD 的联动#

评估不应是孤立的，它应当对应 SDD（Spec-Design-Task）的上下文粒度：

层级对应 SDD评估方法游戏场景示例
L1: 任务级Task代码评估 (Code-based)API 编译通过、单元测试覆盖、配置表 Schema 校验
L2: 设计级Design代码评估 + LLM 裁判是否使用了项目统一的错误处理包？路由定义是否冲突？
L3: 规格级SpecLLM 裁判 + 人工评估NPC 语气是否符合人设？活动逻辑是否会导致刷奖励漏洞？

6.2 集成到 AI Collab Hub & Sandbox#

你不需要单独部署评估平台，Evals 应作为 Pipeline 嵌入现有流程：

  1. Sandbox 内预检 (Self-Correction)
    • Agent 在 Sandbox 完成任务后，自动触发 Code-based Evals。
    • 例子：Agent 写完 Go 代码 -> Sandbox 运行 go vet 和 go test -> 失败 -> 自动要求 Agent 修复 -> 通过后上报 Hub。
  2. Hub 审批流拦截 (Approval Gate)
    • PR 合并前，Hub 调用 LLM-as-a-Judge 进行代码规范审查。
    • 例子：检查代码是否遵循项目命名规范？是否包含硬编码 Magic Number？评分低于 0.8 则驳回 PR，要求人工介入。
  3. 线上监控回流 (Bad Case Loop)
    • 收集线上 Traces（如报错日志、玩家反馈），自动抽取 Bad Case 加入评估数据集，用于下一轮实验。

6.3 实战案例#

案例 A：活动系统代码生成 (Go 后端)#

  • 场景：Agent 生成一个"限时抢购"活动 API。
  • Evals 执行：
    1. Code Eval (硬指标)：protoc 编译成功？go build 通过？(Pass/Fail)
    2. LLM Judge (软指标)：代码是否使用了项目统一的 Redis Lua 脚本扣库存？(Score 0-1)
    3. 结果：两者都 Pass，Agent 才提交 PR。

案例 B：Cocos UI 组件生成 (前端)#

  • 场景：生成一个活动弹窗 UI 组件 (TypeScript)。
  • Evals 执行：
    1. Code Eval：TS 代码无编译错误？cc.Node 节点树结构符合预期？
    2. LLM Judge：按钮点击事件是否绑定了正确的回调？倒计时逻辑是否处理了跨天情况？

案例 C：NPC 剧情对话生成 (策划辅助)#

  • 场景：生成"新手引导 NPC"与玩家的对话。
  • Evals 执行：
    1. Code Eval：输出是否包含禁止词？长度是否在限制内？
    2. LLM Judge：语气是否符合"热情"设定？是否准确引导了操作？
    3. 人工评估：策划抽检 10%，确认文案可用性。

6.4 实施路线图#

  1. 第一阶段 (规范级)：重点建设 Code-based Evals (Lint, Test, Schema)。解决 80% 的"AI 乱写"问题。
  2. 第二阶段 (精益级)：引入 LLM-as-a-Judge 到 Hub 审批流，拦截Architecture级错误。
  3. 第三阶段 (标杆级)：建立自动化回流闭环，SDD 资产与 Evals 联动，实现 AI 自主迭代。
