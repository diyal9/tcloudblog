## 一、游戏公司 AI 落地的真实困境

过去一年，几乎所有游戏公司都在做同一件事：给策划、程序、美术装上 Cursor 或 Claude Code，然后期待产出质量大幅提升。

实际结果呢？**单点提效显著，但生产级交付依然靠人。**

### 1.1 为什么单 Agent 不够？

问题不在于工具能力不足，而在于**游戏开发的本质是长链路、多角色、强依赖的协作工程**。

#### 痛点一：跨模块依赖导致上下文爆炸

一个商业化活动开发涉及：
- 策划输出活动规则文档（Word/Excel，50+ 页）
- 服务端设计数据结构、接口、数据库表（Go/Java）
- 客户端实现 UI 界面、动画、交互逻辑（Cocos/Unity）
- 美术产出 UI 资源、特效、Icon（PSD/PNG/Sprite）
- 配置表导出、资源打表、内存优化
- QA 编写测试用例、回归验证

让 Cursor "帮我做完整个活动"，需要同时塞入几十份文档、上百个代码文件、数百个美术资源的上下文。**LLM 的窗口再大，也装不下一个项目的上下文。**

#### 痛点二：质量标准不一致

Cursor 写的前端代码可能不遵守项目的架构规范；Claude 生成的 Cocos 脚本可能没做对象池复用；Copilot 画的配置表可能漏了必填字段。**单 Agent 无法保证全流程遵守团队既定的工程规范。**

#### 痛点三：失败无法追溯，责任无法定位

一个 Prompt 干完所有事，出了问题不知道是哪个环节错了：是策划的需求理解有误？是前后端接口定义不一致？是美术资源命名不规范？**没有分步验证，调试成本高于手工开发。**

#### 痛点四：无法并行

游戏开发是天然并行的：前端做界面和后端做接口可以同时推进。但单 Agent 是串行执行——它必须先做完 A 才能做 B。**一个 5 天的手工任务，单 Agent 串行下来可能变成 8 天。**

### 1.2 Agent 编排如何解决？

Agent 编排的核心价值不是让 AI 更聪明，而是**让 AI 的开发过程可管理、可验证、可追溯**。

| 痛点 | 单 Agent | Agent 编排 |
|:---|:---|:---|
| 上下文爆炸 | 全部塞进一个 Prompt | 每步只给相关上下文 |
| 质量不一致 | 无约束，自由发挥 | 每步都有规范校验 |
| 无法追溯 | 一锅粥，不知道哪错了 | 节点级日志，精确定位 |
| 串行执行 | A→B→C 依次做 | B 和 C 并行，A 完成后同时启动 |
| 人工介入 | 要么全手动，要么全自动 | 关键节点设置人审关卡 |

**一句话总结：Agent 编排把"一个 AI 干所有事"变成"多个 AI 各管一段，每步可验证，全程可追溯"。**

## 二、主流企业级编排框架对比

### 2.1 LangGraph（Anthropic / LangChain 生态）

**核心机制**：基于有向图的状态机，支持循环、条件分支、人工介入。

```python
from langgraph.graph import StateGraph

graph = StateGraph(AgentState)
graph.add_node("analyze", planning_agent)
graph.add_node("develop", coding_agent)
graph.add_node("review", audit_agent)
graph.add_edge("analyze", "develop")
graph.add_conditional_edges("review",
    lambda s: "deploy" if s["approved"] else "fix")
```

**优势**：
- **图结构原生支持循环和条件分支**，适合复杂的审查-修复循环
- **Checkpoint 机制**支持中断后恢复，适合需要人工审批的节点
- **与 LangChain 生态深度集成**，工具调用、向量检索开箱即用

**劣势**：
- 学习曲线陡峭，需要理解状态图、节点、边、条件路由等概念
- Python 锁定，对 Go/Java/C++ 技术栈团队有语言切换成本
- 长链路调试需要查看完整的状态快照

**谁在用**：
- Anthropic 内部（Claude 的工具调用编排）
- 多家互联网公司的 AI 编程助手（基于 LangChain + LangGraph 构建）

### 2.2 Microsoft AutoGen

**核心机制**：多 Agent 对话式协作，Agent 之间通过互相发消息完成任务。

```python
from autogen import AssistantAgent, UserProxyAgent

coder = AssistantAgent("coder", llm_config=llm_config)
reviewer = AssistantAgent("reviewer", llm_config=llm_config)
user = UserProxyAgent("user", code_execution_config={"docker": True})

user.initiate_chat(coder, message="实现一个活动倒计时组件")
```

**优势**：
- **原生 Docker 沙箱**，Agent 可以直接运行代码并获取输出
- 角色定义简单（UserProxy + Assistant），概念直观
- 多 Agent 自主辩论模式适合代码审查场景

**劣势**：
- **对话可能失控**，Agent 间无硬约束，可能陷入无限循环
- 状态管理弱，执行路径不够透明
- Token 消耗随 Agent 数量和对话轮数线性增长

**谁在用**：
- Microsoft 内部研发工具链
- 多家量化公司和数据分析团队的自动化分析流程

### 2.3 Dify（Dify.AI）

**核心机制**：可视化工作流 + LLM Ops 平台，低代码编排。

**优势**：
- **开箱即用**，提供完整的可视化编辑器，非技术人员也能搭建
- **内置 RAG、工具调用、模型路由**，一站式 LLM 平台
- **后端基于 Go**，高并发场景表现优秀
- 企业级功能完善：API 发布、监控告警、权限管理

**劣势**：
- **定制能力受限**，可视化编排难以表达复杂的循环、动态分支
- **平台锁定风险**，工作流定义依赖平台 DSL，迁移成本高
- **部署复杂**，需要 Redis、PostgreSQL、向量数据库等多个组件

**谁在用**：
- 多家金融、电商、客服公司的 AI 应用平台
- 非技术团队的 AI 工作流快速搭建

### 2.4 对比矩阵

| 维度 | LangGraph | AutoGen | Dify | 自研 DAG |
|:---|:---|:---|:---|:---|
| 编程范式 | 图状态机 | 多 Agent 对话 | 可视化工作流 | DAG 拓扑排序 |
| 语言 | Python | Python | Go + Python | Go |
| 灵活性 | ★★★★★ | ★★★★ | ★★★ | ★★★★ |
| 易用性 | ★★ | ★★★ | ★★★★★ | ★★★ |
| 循环/分支 | ✅ 原生 | ⚠️ 易失控 | ✅ 可视化配置 | ✅ 条件路由 |
| 人工介入 | ✅ Checkpoint | ⚠️ 需手动 | ✅ 内置审批 | ✅ 条件分支 |
| 部署复杂度 | 低 | 中 | 高 | 低（单二进制） |
| 适合团队 | AI 工程团队 | 研究/实验 | 业务/产品 | 基础设施团队 |

## 三、手游公司的落地方案

### 3.1 选型决策

对于手游开发团队，我们的选择标准是：

1. **语言栈一致**：游戏服务端通常是 Go/Java/C++，引入 Python 编排层意味着增加运维负担
2. **部署轻量化**：游戏公司已有服务器集群（匹配服、战斗服、大厅服），不希望额外部署一整套 LLM 平台
3. **可定制化**：游戏开发的工作流高度定制（打表流程、资源管线、多语言本地化），可视化平台难以覆盖
4. **与引擎集成**：需要与 Cocos/Unity 编辑器、打表工具、资源服务器深度集成

因此，我们选择**自研轻量级 DAG 编排引擎**（Go 实现），原因如下：
- 单二进制部署，与现有游戏服务端基础设施一致
- DAG 拓扑排序天然支持并行节点（前后端同时开发）
- 条件分支支持审查-修复循环
- 可直接调用游戏项目已有的打表脚本、资源打包工具

### 3.3 各角色在人回路中的定位

#### 策划（Game Designer）

**传统工作流**：写活动文档 → 拉会对齐 → 等前后端开发 → 验收 → 反复改

**AI 编排工作流**：
1. 策划提交需求文档（Word/Notion）
2. **需求分析 Agent** 自动拆解为：活动规则、数据结构、接口列表、UI 清单、资源配置表
3. 策划**审核拆解结果**（人审关卡），确认无误后批准进入开发
4. 开发完成后，策划**验收产出**，不通过则触发返工

**策划的角色变化**：从"写文档的人"变成"审核需求拆解和验收结果的人"。AI 负责把模糊的需求变成精确的技术规格，策划只负责确认"这就是我想要的"。

#### 程序（Client/Server Developer）

**传统工作流**：接需求 → 写代码 → 联调 → 修 Bug → 提交 Code Review

**AI 编排工作流**：
1. 程序**定义架构约束**（通过 `.cursorrules` 或项目规则文件）：分层规范、命名约定、性能红线
2. **开发 Agent** 在约束下自动生成代码
3. **审查 Agent** 自动运行代码质量检查（lint、静态分析、单元测试）
4. 程序**Review AI 的代码**，重点检查架构决策和边界情况
5. 通过后自动合并，触发下游节点

**程序的角色变化**：从"写代码的人"变成"制定规则和审查 AI 产出的人"。AI 负责重复性编码，程序负责确保代码质量和架构一致性。

#### 美术（Artist）

**传统工作流**：接需求 → 设计 → 产出资源 → 命名 → 上传服务器 → 客户端引用

**AI 编排工作流**：
1. **美术 Agent**（集成 Midjourney/Stable Diffusion）根据策划文档自动生成 UI 草图和 Icon
2. 美术**精修 AI 产出**，调整细节、风格一致性
3. 自动执行资源导出（切片、压缩、打 Atlas）、命名规范校验
4. 资源上传 CDN，**配置 Agent** 自动更新资源引用表

**美术的角色变化**：从"从零开始画"变成"AI 出草图，美术精修定稿"。产出效率提升 3-5 倍，且资源命名和格式零出错。

#### 运营（Operator）

**传统工作流**：配置活动参数 → 配表 → 等 QA 验证 → 上线 → 监控数据

**AI 编排工作流**：
1. 运营**设定活动目标**（留存、付费转化、DAU）
2. **配置 Agent** 根据目标自动生成活动参数表（奖励倍率、难度曲线、付费点）
3. **QA Agent** 自动运行配置校验（奖励溢出检查、数值平衡验证）
4. 运营**审核配置结果**，通过后自动上线
5. 上线后**监控 Agent** 持续追踪数据，异常时自动告警

**运营的角色变化**：从"配表和等验证的人"变成"定义目标和审核结果的人"。AI 负责数值计算和自动化验证。

### 3.4 完整案例：商业化活动全 AI 管线

以"限时充值返利活动"为例，完整的编排流程如下：

#### Step 1：需求拆解（自动化，策划审核）

策划提交活动文档后，需求分析 Agent 自动拆解：

- **活动规则**：累计充值 100 元返利 10 元，500 元返利 80 元，1000 元返利 200 元
- **技术需求**：
  - 服务端：新增充值返利接口、累计金额统计、防刷逻辑
  - 客户端：活动页面 UI、充值入口、返利弹窗、进度条动画
  - 配置表：活动参数表（时间、档位、返利值）
  - 美术资源：活动 Banner、按钮 Icon、进度条素材

策划确认拆解无误，批准进入开发。

#### Step 2：并行开发（自动化）

DAG 引擎同时启动三个并行节点：

| 节点 | 执行者 | 输入 | 输出 |
|:---|:---|:---|:---|
| 服务端开发 | 后端 Agent | 接口列表、数据结构 | API 代码、数据库迁移脚本 |
| 客户端开发 | 前端 Agent | UI 清单、交互说明 | Cocos 组件代码、Prefab |
| 美术资源 | 美术 Agent + 人工精修 | 风格参考、尺寸要求 | PNG 资源、Sprite Atlas |

前端和后端同时开发，互不阻塞。

#### Step 3：联调测试（自动化，程序审核）

Merge 节点等待三个并行节点全部完成：

1. **配置 Agent** 自动合并前后端接口定义，生成 Mock Server
2. **测试 Agent** 自动运行接口联调测试
3. 生成联调报告，程序 Review 后批准

#### Step 4：代码审查（自动化）

**审查 Agent** 自动执行：
- 代码规范检查（lint、命名约定）
- 安全扫描（SQL 注入、XSS、反作弊）
- 性能检查（DrawCall 分析、内存泄漏检测）
- 单元测试覆盖率验证

审查通过 → 进入部署；不通过 → 自动触发返工节点。

#### Step 5：配置验证（自动化，运营审核）

**配置 Agent** 生成活动参数表后，自动运行：
- 数值平衡验证（返利倍率是否在合理区间）
- 奖励溢出检查（极端情况下是否会导致经济系统崩溃）
- 时间窗口校验（活动开始/结束时间是否冲突）

运营审核配置表，确认后上线。

#### Step 6：部署上线（自动化）

- CI/CD 自动构建客户端和服务端
- 灰度发布（先 10% 玩家）
- 监控 Agent 持续追踪核心指标
- 异常自动告警，必要时自动回滚

#### 时间对比

| 阶段 | 传统人工 | AI 编排 | 节省 |
|:---|:---|:---|:---|
| 需求拆解 | 1 天（人工+对齐会） | 2 小时（AI 拆解+人审） | 85% |
| 服务端开发 | 2 天 | 4 小时（AI 编码+人审） | 75% |
| 客户端开发 | 3 天 | 6 小时（AI 编码+人审） | 75% |
| 美术资源 | 2 天 | 6 小时（AI 出图+精修） | 75% |
| 联调测试 | 2 天 | 2 小时（自动化测试） | 88% |
| 代码审查 | 1 天 | 1 小时（自动审查） | 88% |
| 配置验证 | 1 天 | 30 分钟（自动校验+人审） | 90% |
| 部署上线 | 0.5 天 | 10 分钟（自动化） | 95% |
| **总计** | **12.5 天** | **约 2.5 天** | **80%** |

**注意**：这 2.5 天中包含人工审核时间（策划审核需求、程序审查代码、运营确认配置）。纯 AI 执行时间约 8 小时，但关键环节保留人工审核是必要的安全阀。

## 四、避坑指南

### 4.1 不要一开始就搞全自动化

先让 AI 做辅助（需求拆解、代码生成），人做审核。等每个节点的质量稳定后，再逐步减少人工介入。

### 4.2 图结构比对话更可控

AutoGen 的多 Agent 对话看似自然，但生产环境容易失控。优先选择**显式的图结构（DAG）**，执行路径可预测、可调试。

### 4.3 规范文件是质量保障的核心

AI 编码的质量取决于约束的精确度。必须为每个 Agent 定义清晰的规则文件：
- 架构约束（分层规范、接口契约）
- 编码规范（命名、注释、错误处理）
- 性能红线（GC 限制、DrawCall 上限、内存预算）
- 安全底线（输入校验、反作弊检查）

**规则文件写得越精确，AI 产出越可靠。**

### 4.4 保留人工审核的关键节点

以下节点必须保留人审：
- 需求拆解结果（策划确认"这就是我想要的"）
- 代码架构决策（程序确认架构合理性）
- 数值平衡配置（运营确认经济系统安全）
- 灰度发布决策（确认是否扩大灰度范围）

## 五、总结

对游戏公司而言，Agent 编排不是让 AI 替代开发者，而是**让 AI 承担可标准化的工作，让人专注于决策和审核**。

单 Agent（Cursor/Claude Code）解决的是"写代码快不快"的问题，而 Agent 编排解决的是**"一整个活动从需求到上线能不能标准化、可管理、可追溯"的问题。**

选型建议：
- **Python 团队**：LangGraph 是最佳选择，灵活度高，社区活跃
- **Go/Java 团队**：自研轻量级 DAG 引擎最务实，与现有基础设施一致
- **非技术团队主导**：Dify 可视化工作流最快落地

但无论选什么，核心原则不变：**从简单开始，图结构优先，保留人审，持续迭代。**
