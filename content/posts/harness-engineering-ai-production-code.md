## 原文：Harness 工程化实践

今天聊一个很多人都在问的问题：怎么让 AI 写出来的代码真的能上线？

你可能已经用了很多代码 Agent 写项目，体感上确实快了。但如果你在一个真实的企业级项目里，面对十几万行代码、RPC 框架、流程引擎、配置中心、分布式缓存全套中间件，你大概率遇到过这个情况：**Agent 写出来的代码，语法完美，风格统一，但业务逻辑有微妙的错误。**

这种错误在编译层面没问题，但是实际运行却错漏百出。为了修这些错误，形成恶性循环，不仅耗费金钱，还让代码仓库变成了一座屎山。
- 比如价格字段用了 `double`，而不是 `long`，单位搞错了。
- 比如改了主链路但漏了国际化链路的同步修改。
- 比如调用外部服务没设超时和降级。

这些错误有一个共同特点，它们不是模型不够聪明导致的。模型的单步推理能力已经很强了。**问题出在模型不知道你项目里那些从来没被写下来的规矩。**

### 阿里团队实践：从 25% 到 90%

今天分享一下阿里团队的一个实践：他们在一个存量 Java 项目上，用一周时间搭了一套 **Harness 体系**，把 AI 代码率从 25% 拉到了 90%。

不是换了更强的模型，仅仅是根据 Harness Engineering 的定义，搭了一套**外部的约束和反馈系统**。

今天我就来拆这个实践。不讲概念，讲操作，你听完之后能直接在自己的项目里开始搭。

### 三次 AI 工程实践跃迁

先说一个背景：AI 工程实践正在经历第三次范围跃迁。

1. **Prompt Engineering (2022-2024)**：核心是「怎么写好一条指令」。Few-shot、Chain-of-Thought、角色设定，都是在优化单次交互。
2. **Context Engineering (2025)**：核心是「给 Agent 看什么」。动态构建上下文窗口，Agentic RAG 检索，工具定义。Shopify CEO 的比喻是「给邮件附上所有正确的附件」。
3. **Harness Engineering (2026)**：核心不再是一次对话或一次上下文窗口，而是**设计跨越多个会话、多个 Agent 角色、多个执行阶段的完整系统Architecture**。

本质上来说，由于大模型的不确定性，每次你发现 Agent 犯了一个错误，你就需要花时间工程化地消除它再次发生的可能性。**不是修 Prompt，不是加一句「请注意」。是工程化地消除。** 把约束变成文件、规则、自动化检查，成为系统的一部分。

### Agent 的四种典型失败模式

为什么需要这么做？因为 Agent 有四种典型的失败模式：

1. **试图一步到位**：拿到复杂需求后，Agent 倾向于在一个上下文窗口里完成所有工作，上下文填充率超过 40% 之后，输出质量快速衰退，开始出现幻觉、循环输出。
2. **过早宣布胜利**：Agent 完成部分工作就说「编码完成」，但实际上编译都过不了。
3. **没做端到端验证就标记完成**：Agent 认为功能实现了，但没跑过完整的测试，部署后才发现关键路径不通。
4. **冷启动问题**：每次新会话都要花大量 token 重新理解项目结构，真正用于编码的预算被严重挤压。

这四种失败模式的共同根源是：**Agent 缺乏外部的结构化约束和反馈机制。** 你不能指望 Agent 自己审查自己。

### Harness 体系核心组件

实操部分。这个团队的做法是在项目根目录下建一个 `.harness` 目录，里面放四样东西：

#### 1. 规则体系 (.harness/rules)
告诉 Agent「标准是什么」。三份文件：工程结构、开发流程规范、项目编码规范。
- **工程结构**：分模块、职责、分层（Controller/Service/DAO）。
- **编码规范**：价格用 `long` (单位为分)，外部调用设超时/降级，流程组件不写大段逻辑。
这些规则以前靠口头传承，现在写成文件让 Agent 执行。

#### 2. 技能体系 (.harness/skills)
九个 Skill，每个是一份 SOP。
- **分层编码规范 Skill**：八份模板，Agent 按规范一层一层实现。
- **评审 Skill**：定义计划评审和执行评审，意见包含描述、建议、优先级。
- **Unit Test Skill**：改动驱动测试，优先用线上真实出入参构造数据。

#### 3. 知识库 (wiki/)
链路梳理、数据模型、核心业务流程。注意：**按需获取，不是全量灌入**。

#### 4. 变更管理 (.harness/changes)
全流程留痕：需求分析 → 任务拆分 → 编码报告 → 评审记录 → 部署验证。文件版本化，永不删除。

### 核心引擎：Application Owner Agent

这四样东西需要一个大脑来串联。团队定义了一个 `ApplicationOwner` Agent 角色，包含五个模块：
1. **角色与背景**：20-30 行，提供够用视野。
2. **配置中枢缩影**：结构化表格，快速定位知识。
3. **七项核心职责**：需求理解、任务拆解、分发、验收、文档管理、知识回答。
4. **工作流程**：十个阶段的调度逻辑。
5. **沟通原则与硬性约束**：Must do / Don't do 清单。

**本质**：将资深开发者的工作习惯和决策逻辑编码化。

### 十阶段开发流水线

1. 需求分析
2. 需求审批
3. 编码实现
4. 编码评审
5. 单测编辑
6. 单测评审
7. 代码推送
8. CI 验证
9. 部署验证
10. 用户确认

#### 关键设计
- **精确回退**：失败路由回修复环节，非从头再来。例如 CI 失败回退到单测编辑。
- **防循环机制**：审批限制循环次数（如 3 轮），超时升级人工。
- **分离执行与评判**：编码 Agent 负责执行，评审 Agent 负责评判，避免自我验证盲区。

### 效果与总结

这套体系上线后，AI 代码率从 25% 提升到 90%，质量可控，返工大幅减少。

Harness 的价值在于**让 Agent 的错误可控、可发现、可修复**。通过工程化的约束和反馈，把隐性知识显性化，实现 AI 代码的规模化生产和质量保障。

## 总结与分析

### 核心观点
Harness 体系的核心是**将隐性知识显性化**，通过构建外部约束系统，解决 Agent 在复杂项目中“好看不好用”的问题。它标志着 AI 工程实践从 Prompt/Context 时代进入了系统Architecture时代。

### 失败模式分析
| 失败模式 | 根源 | Harness 解法 |
| :--- | :--- | :--- |
| 试图一步到位 | 上下文过载 | 十阶段流水线拆分 |
| 过早宣布胜利 | 缺乏自我评估 | 质量门禁 + CI 验证 |
| 未端到端验证 | 流程缺失 | 强制部署验证 + 单测评审 |
| 冷启动浪费 | 无结构上下文 | 按需获取知识库 + 角色定义 |

### 实施路线图

#### 1. 最小可行方案 (MVP)
- 编写**编码规范文件**（核心规则）。
- 定义**评审 Skill**（评审 SOP）。
- 配置**CI 门禁**（自动化检查）。

#### 2. 扩展实施
- 完善**技能体系**（分层模板、测试指南）。
- 部署**Application Owner Agent**。
- 建立**知识库按需检索机制**。

#### 3. 持续优化
- 监控失败模式，更新规则。
- 沉淀业务知识到文档。
- 优化流水线回退逻辑。

Harness 体系不仅仅是工具的堆叠，更是**工程纪律的数字化**。对于希望规模化使用 AI 的团队来说，这是必经之路。
