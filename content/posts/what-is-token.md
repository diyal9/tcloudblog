# 🧠 什么是 Token？

**Token** 是 OpenAI 类模型在内部处理文字时的基本切分单位。

它不是「一个字」，也不是「一个词」。同一段自然语言，在人眼看起来是一句话，在模型眼中则是一串 Token。

## 💻 Token 是怎么数的？

Token 的粒度可以短到一个字符，也可以长到接近一整个常见英文单词。空格、标点、以及单词被拆开后的「半拉子」，都可能占据一个 Token。

### 🌐 真实世界的 Token 规模
通过一些例子，我们可以建立直觉：

* 韦恩·格雷茨基的名言 *"You miss 100% of the shots you don't take"* → 约 11 个 Token
* 《OpenAI 宪章》全文 → 约 476 个 Token
* 《美国独立宣言》 → 约 1,695 个 Token

可以看出，名著级段落本身就已经是上千 Token 量级；再加上系统提示、工具返回、多轮历史对话，很容易就撞上下文限制。

## 💰 Token 与 API 成本

### 为什么同样的「字数」费用差别这么大？

因为 Token 的粒度定义不同导致：
* 英文文本通常会有更高的 Token/字符比。
* 「看似微小的空格与大小写」有时会影响 Tokenization。
* Prompts 的排版方式、模型版本等都可能影响费用和上下文预算。

在 API 响应中，用量会按类别拆分统计：
1.  **输入 Token**：你的请求里消耗的 Token。
2.  **输出 Token**：模型生成内容里的 Token。
3.  **缓存 Token**：对话历史中被复用、按缓存计费的部分。
4.  **推理 Token**：部分高级模型内部的「思考」步骤，会产生额外 Token。

---

## 🌍 Token 与上下文预算

每个模型都有一个 Token 的上下文预算（输入 + 输出）。
新增代际的高容量模型可以到达数十万 Token 量级，但具体数字受模型版本、账号档位等因素影响。

### 🔺 撞上下文上限怎么办？

若提示或回复过长导致 Token 数量超标，可以参考以下技巧：
* **缩短或改写**：精简系统提示与用户提示。
* **切块 (Chunking)**：把长文档切块，分段检索或摘要。
* **预处理**：先发摘要，再发细节，或对输入做压缩。

## 📊 英文 Token 数的估算

对于英文，根据官方经验法则有以下估算方式（仅供参考）：

| 关系 | 大致比例 |
| :--- | :--- |
| 1 个 Token | ≈ 4 个字符 |
| 1 个 Token | ≈ ¾ 个单词 |
| 100 个 Token | ≈ 75 个单词 |
| 1–2 句 | ≈ 30 个 Token |
| 1 段 | ≈ 100 个 Token |
| 1,500 个单词 | ≈ 2,048 个 Token |

> **注意**：这些估算方法可以在写英文 Prompt 或成本预估中用于数量级判断，但无法替代精确计费。

---

## 🛠️ 进阶理解：同一个词，Token 可能不同

例如单词 `red`：
* ` red`（前有空格、小写）与 ` Red`（前有空格、首字母大写）是不同 Token。
* 句首的 `Red`（不带前导空格）又可能是第三种切法。

这提醒我们：API 里看似微小的细节，可能会影响 Tokenization，进而影响费用和上下文限制。

---

## 🎯 小结

1. **基本单位**：Token 是模型内部处理文本的基本单位，不是字数或词数。
2. **精确估算**：英文可借助经验法则粗估，非中文需更谨慎。
3. **分类计费**：API 中要区分输入、输出、缓存、推理等 Token 类别。
4. **工具辅助**：精准计数建议使用 Tokenizer 工具或 `tiktoken` 库。
