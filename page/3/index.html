graphify代码图谱的使用

Andrej Karpathy 曾提出过一个痛点：我们都有一个塞满代码、论文、推文截图的 /raw 文件夹，但这些素材之间的关联全在脑子里。AI 助手想要理解它们，就必须从头读取所有文件，消耗大量 Token。 今天介绍的 Graphify，就是为了解决这个问题而生的 AI-Programming助手「技能插件」。 Graphify 能将任意文件夹中的代码、文档、论文、图片转化为一个可查询的知识图谱。更厉害的是，构建图谱后，AI 查询仅消耗原始 Token 的 1/71.5。 Graphify 的核心能力 1. 双通道提取引擎 (Dual-Channel Extraction) Graphify 的提取过程分为两个并行的通道，确保速度与深度的平衡： 通道 A — AST 确定性提取：对代码文件使用 Tree-sitter 进行语法树分析。 优势：零 LLM 开销，速度极快。 内容：提取类定义、函数签名、导入关系、调用图以及带有 # WHY: 等标记的设计决策注释。 支持语言：Python, TS/JS, Go, Rust, Java, C/C++ 等 15 种主流语言。 通道 B — 语义提取：由 LLM 子代理对文档（.md/.txt）、论文（.pdf）和图片进行概念抽取。 内容：图片分析、PDF 引用挖掘、核心概念提取。 2. 诚实的三级置信度体系 这是 Graphify 区别于普通图谱工具的一大亮点。每条关系边都被严格标记： EXTRACTED：从源码直接找到的显式关系（置信度 1.0）。 INFERRED：合理推断的关系（置信度 0.6-0.9），例如两个模块虽然没有直接调用，但共享了数据结构。 AMBIGUOUS：不确定的关系，留待人工审查。 这种设计让 AI 明确知道自己是在引用事实还是在推测，大幅降低了幻觉风险。 3. 图拓扑社区检测 Graphify 不使用向量数据库，而是基于图的边密度拓扑，使用 Leiden 算法进行社区发现。这意味着它能自动识别出代码库中真实的模块边界，哪怕它们在文件系统中是混杂在一起的。 ...

Code-Knowledge-Graph对比：GitNexus和Graphify

AI-Programming工具已经越来越会写代码了，但在真实工程里，更关键的问题其实是：AI 是否真的理解代码？围绕这个问题，Code-Knowledge-Graph开始变得重要。 GitNexus 和 graphify 都在做“Code-Knowledge-Graph”，但它们的方向并不一样： GitNexus: 更像是给 AI-Programming工具加一层工程基础设施，让 AI 在修改代码前知道依赖关系、了调用链和影响范围。 graphify: 更像是把代码、文档、图片、视频等资料整合成一个知识网络，帮助人快速理解一个系统。 简单说：GitNexus 偏执行，graphify 偏理解。 GitNexus：让 AI 改代码前先看懂影响范围 GitNexus 的定位很明确：它会把代码库索引成一个知识图谱，覆盖依赖关系、调用链、聚类、执行流等信息，然后把这些能力暴露给 AI agent 使用。 它的 README 中提到，GitNexus 是在为 agent 构建上下文的“神经系统”。这句话其实很准确。 很多 AI-Programming工具在写代码时，最大的问题不是不会生成代码，而是容易“漏看上下文”。比如： 改了一个函数，却不知道哪里调用了它； 修改了一个类，却忽略了继承关系； 重构一个模块，却没有意识到影响了其他仓库； 做 PR review 时，只看到了 diff，看不到完整影响面。 GitNexus 要解决的就是这些问题。它不是简单地画一张代码结构图，而是把代码库变成一个可查询、可分析、可执行的结构系统。 GitNexus 的工程特点：结构优先，面向修改 从实现上看，GitNexus 更偏“结构化分析”。它会使用 Tree-sitter 做 AST 解析，并进一步分析： import 关系； 函数调用； 类继承； 构造器推断； self/this 接收者类型； 跨文件引用； 执行流； 影响范围。 此外，它还提供混合搜索能力（BM25、语义检索和 RRF 排序组合），并支持本地运行，强调隐私和无网络依赖。 对大型代码库来说，这类能力非常关键。因为大型项目的问题从来不是“有没有代码”，而是代码之间的关系太复杂。GitNexus 的价值在于，它把代码库提前索引成一个可查询的结构图，让 AI 和工程工具能够在此基础上做判断。 GitNexus 不只是看图，而是进入工程流程 GitNexus 真正工程化的地方，在于它不是停留在“可视化”。它提供了一些面向真实开发流程的能力，例如： ...
